IroHarness Inspiration Map

参考にするが、人格の所有権はIroHarnessに残す。

CursorTuberKit、Neuro SDK、AIAvatarStackChanを比較し、IroHarnessへ何を吸収し、 どこを別リポジトリに分けるべきかを整理した公開用マップ。

結論

まずはIroHarness本体1リポジトリで公開する。Firmware、Unity、Godotなど ツールチェーンが分かれるものだけ、後から別リポジトリ化する。

3つの参考プロジェクト

CursorTuberKit

Browser avatar and live-stream workflow

  • ブラウザファーストの配信アバター体験
  • SVGとMotionPNGTuberの切り替え
  • VOICEVOXと再生同期リップシンク
  • YouTube、Twitch、Kickコメント入力
  • 発話が重ならない直列キュー

Neuro SDK

Game and app integration API

  • 外部アプリ向けの明確なAPI
  • 公式SDKとコミュニティSDKの分離
  • 統合テスト用のテストbot
  • 高頻度操作は低レベル制御へ逃がす設計

AIAvatarStackChan

Physical avatar and device stack

  • サーバーとデバイスFirmwareの分離
  • STT、LLM、TTSの差し替え
  • VAD、PTT、ミュート、音量、タッチ
  • カメラ画像を送る視覚トリガー
  • ハードウェア抽象化

リポジトリ戦略

今は本体1リポジトリがよい

IroHarnessの価値は、1人格、複数body、複数micro harness、Project OS、 audience permissionsを同じ境界で扱えること。最初から分けると思想が伝わりにくい。

後で分ける候補

iroharness-stackchan-firmware PlatformIO / Arduino / M5Stack向け。Firmwareのリリース周期が違うため。
iroharness-unity-sdk Unity Packageとして配布したくなったタイミングで分離。
iroharness-godot-sdk Godot利用者向けのSDK、サンプル、asset構成が固まったら分離。
iroharness-community-adapters 外部コントリビュータのadapterが増え、本体とリリース周期を分けたくなったら分離。

IroHarnessの非交渉ライン

人格

人格はmacro harnessが持つ。モデル、SDK、デバイス、外部harnessは人格そのものではない。

状態

長期作業はProject OSに残す。会話ログだけを正本にしない。

権限

ファン、開発者、管理者で許可される操作を変える。人格は変えない。
